## Previous Request
**Status:** DONE
**Date:** 2026-04-25
**Request:** Phase 8.6.3 — Fix: layout's SG never reached the FP generator (contract mismatch).
**Progress:** Layouts contained `has_staffelgeschoss=True` but the user still saw no SG in the rendered floor plans. Root cause: contract mismatch between the two optimizers. **Layout side:** `BuildingFootprint.stories` = LOWER count, SG is a +1 bonus when `has_staffelgeschoss=true`. **FP generator side:** `FloorPlanRequest.stories` = TOTAL count, then subtracts 1 internally to make the top one a Staffelgeschoss when `enable_staffelgeschoss=true`. So when `optimize.py` passed the layout's `stories` (lower count) directly into the FP request along with `enable_sg=true`, the FP gen produced `(lower − 1)` regular floors + 1 SG = 1 floor *fewer* than the layout chose. SG appeared once but was the LAST of the lower floors getting reclassified as SG, so visually the user saw "lower stories - 1, plus a staffelgeschoss". With `lower=1` it produced 0 regular + 1 SG, which probably looked like nothing changed. Fixed by translating in `_generate_floor_plans_for_layout`: `fp_total_stories = building.stories + (1 if has_staffelgeschoss else 0)` before passing into `FloorPlanRequest.stories`. End-to-end smoke now correct: layout picks 3 lower + SG (H=12.2m) → FP request stories=4, enable_sg=true → FP gen emits 3 regular floors (ground + 2 standard) + 1 SG floor with correctly inset grid (32.1×12.2 vs 36.1×14.7 lower). Backend: 68 tests pass.

## Previous Request
**Status:** DONE
**Date:** 2026-04-25
**Request:** Phase 8.6.2 — SG as a bonus floor + tightened §6 reject threshold.
**Progress:** 8.6.1 corrected the §6 envelope math but over-corrected the chromosome semantics: it stole one storey from the budget when SG was on, which made SG and no-SG variants produce the same total NFA at the same gene — so the GA had no reason to ever pick SG (the no-SG variant always wins on the gradient). 8.6.2 reverts that and treats SG as a *bonus* floor sitting on top of the chromosome's full story count. BauNVO doesn't count SG as a Vollgeschoss when its footprint is < 75 % of the floor below (our 2 m symmetric setback satisfies that for typical Goldbeck dimensions), so SG genuinely adds one effective floor of NFA without consuming a slot in `max_stories`. The §6 math still works because the SG facade is inset by `setback_m` (2 m) while its required ground projection grows by only `Δd ≈ 0.4 × 3.05 = 1.22 m` — net result: the SG envelope sits 0.78 m INSIDE the lower envelope on every face, so adding SG never tightens the plot fit. Tightened the §6 hard-reject threshold from 30 % overflow → 5 % so the final emitted layouts cleanly fit the parcel. Also steepened the soft penalty from `r²` (with `r = 1 − overflow_ratio`) to `(1 − 8·overflow_ratio)²` so the GA's gradient pushes harder toward zero overflow. Multi-plot smoke test (40×60 max-5, 25×35 max-3, 30×40 max-4): every emitted layout has SG, every §6 envelope fits the plot perfectly (0 violations, 0.00 % max overflow). Top result on 40×60: 4-storey + SG (5 effective floors), 49 units, 3670 m² GFA, vs. 3-storey-no-SG baseline of ~28 units / 2147 m². Backend: 68 tests pass.

**Request:** Phase 8.6.1 — Fix: SG never appearing in layouts.
**Progress:** Two real bugs were preventing the GA from ever picking SG. **(1) Story-budget overflow.** `decode_gene` and `_decode_buildings` returned `stories` (lower count) independently of the `has_staffel` flag, then SG added one more storey on top. A chromosome with `stories=5` + `has_staffel=true` produced a 6-storey effective building whose §6 envelope (`H_total = 18.3 m → d = 7.32 m`) almost always overflowed the plot — so SG layouts systematically failed the §6 ⊂ plot check while non-SG didn't. Fixed by treating the chromosome's stories gene as the *total* story budget when SG is enabled: lower stories becomes `raw_stories − 1`, total height stays equal to a no-SG layout with the same gene. SG only activates when `raw_stories ≥ 2` (a 1-storey building can't have a penthouse over zero floors). Smoke check: SG=on with raw=5 → stories=4+SG; raw=2 → stories=1+SG; raw=1 → stays 1 with SG disabled. **(2) Zero fitness reward for SG.** `_estimate_units` computed NFA as `footprint × stories × 0.85`, ignoring the SG floor entirely. Same in `_evaluate`'s `total_gfa` calculation. Result: SG could only fail the §6 check, never gain reward — the GA learned "no SG = better". Fixed both: NFA and GFA now add `(W − 2·setback) × (D − 2·setback) × 0.85` for the SG floor when applicable. End-to-end smoke test on a 40×60m plot with default regs produced SG layouts ranking in the top-5 (e.g. a 2-storey + SG variant alongside three pure 3-storey variants). Backend: 68 tests still pass.

# Project Architecture

## Quick Start
```bash
# Backend (MUST run from backend/ directory)
cd backend && uvicorn app.main:app --reload --host 0.0.0.0 --port 8000

# Frontend
pnpm dev
```

## Backend (`/backend/app/`)
- **Generator**: `services/floorplan/goldbeck_generator.py` — 7-phase deterministic layout engine
- **Optimizer**: `services/floorplan/optimizer.py` — Genetic algorithm, 17-criterion fitness
- **Quality Scoring**: `services/floorplan/quality_scoring.py` — 5 advanced architectural criteria
- **Constants**: `services/floorplan/goldbeck_constants.py` — All Goldbeck system dimensions
- **Rules**: `services/floorplan/architectural_rules.py` — Building code validation
- **IFC Export**: `services/ifc_exporter.py` — IFC 2x3 with IfcOpeningElement void cutting
- **Cadastral**: `services/cadastral.py` — 16-state parcel lookup (Nominatim + WFS + Overpass + synthetic)
- **Regulations**: `services/german_regulations.py` — German building code parameters
- **WebSocket**: `services/ws_sync.py` — Live-sync broadcaster for Rhino/GH
- **Models**: `models/floorplan.py` — All Pydantic data models

## Frontend (`/frontend/`)
- React/TypeScript with Three.js 3D building viewer
- `components/three/building-detailed.tsx` — 3D renderer with geometric wall matching

- `components/map/cadastral-map.tsx` — Mapbox GL map with parcel selection
- `hooks/use-floor-plan.ts` — Generation hook (passes Staffelgeschoss flag)

## Key Technical Notes
- Backend must run from `/backend/` dir or you get ModuleNotFoundError
- Don't use `--reload` during optimization (causes job loss)
- CORS configured for ports 3000-3002 on localhost and 127.0.0.1
- `BuildingFloorPlans.building_width_m` = generator's `length_m` (long facade)
- `BuildingFloorPlans.building_depth_m` = generator's `depth_m` (short dimension)
- Plan coords (x, y) → Three.js (x, y_up, -y_plan). Wall rotation uses `angle` not `-angle`
- Per-floor `StructuralGrid` has own `building_length_m`, `building_depth_m`, `origin` for Staffelgeschoss
- Bay widths: 3.125, 3.75, 4.375, 5.00, 5.625, 6.25m
- IFC walls placed at `wall.start` with local X along wall direction, profiles extruded along Z
